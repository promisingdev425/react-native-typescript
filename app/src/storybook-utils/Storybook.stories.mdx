# What is Storybook?!

Storybook helps you build UI components in isolation from your app's business logic, data, and context.
That makes it easy to develop hard-to-reach states. Save these UI states as **stories** to revisit during development, testing, or QA.

We recommend building UIs with a [**component-driven**](https://componentdriven.org) process starting with atomic components and ending with pages.

## Writing Stories

We write our stories in [CSF format](https://storybook.js.org/docs/react/api/csf)
which enables automatic documentation of the component by reading the
TypeDoc blocks and Typescript types of your code. This format gets shared
between both our native and web storybooks.

Here are some things you should keep in mind in order to write useful docs:

#### Describe your component

Your component files should include TypeDocs that describe the
purpose of the component and highlight the main props available.

```tsx
/**
 * Describe your component. What does it do? What are the main props?
 */
export const MyComponent = () => {}
```

#### Type and comment your props

Typing your component props allows the automatic documentation
to determine how users can interact with the props for your component.
Don't forget to document them as well!

```tsx
interface MyComponentProps {
  /**
   * Describe each prop your component accepts
   */
  foo: string
  /** You can use a shorthand like this too */
  bar: boolean
}
export const MyComponent = ({ foo, bar }: MyComponentProps) => {}
```

#### Export a Template story

After the default export in your "stories" file, you should
export a `Template` component that spreads its props onto your component.
This allows the arguments table of the documentation to live edit your
component.

```tsx
export default {...}

// In your story file, the first story after the default
// export is used as the primary story for the page.
export const Template = (props) => <Text {...props} />;
Template.storyName = 'Text';
```

#### Document component variations

If your component has multiple variations, it's good practice to export additional
stories from your "stories" file that show these variations. This helps developers understand
what variations are available at a glance.

#### ArgTypes

Use [ArgTypes](https://storybook.js.org/docs/react/api/argtypes)
to customize the autogenerated props for your component.
There are some good examples of the control types available
for use with ArgTypes in the following repo:

https://github.com/storybookjs/react-native/tree/next-6.0/examples/native/components/ControlExamples

#### MDX Documentation

If the auto generated docs don't fit the bill, you can write long form
documentation by using an [MDX file](https://storybook.js.org/docs/react/api/mdx)
instead of a CSF story file. This allows you to write arbitrary markdown
to describe your component.

#### MDX with CSF

If you want to combine the CSF and MDX file formats, you can import your MDX file
into your CSF file as described here:

https://github.com/storybookjs/storybook/blob/next/addons/docs/docs/recipes.md#csf-stories-with-arbitrary-mdx

If you go this route, you might want to use the following as a starting point
for your MDX file so it closely matches the default results:

https://github.com/storybookjs/storybook/blob/next/addons/docs/docs/docspage.md#remixing-docspage-using-doc-blocks
